The Manifesto

Six principles.

What spec-driven agentic development actually requires — not aspirations, working discipline.

  1. 01

    Knowledge, not code.

    A library locks framework, ORM, dialect, runtime. A spec describes only what is invariant — data model, flows, threats, acceptance. The agent decides the rest. When your stack changes, the spec doesn’t.

  2. 02

    Stack-agnostic by discipline.

    Every spec is structured in three layers: semantic (what + why, concrete), mechanism (how, abstract — agent decides), illustrative (reference snippets, concrete but marked). A spec passes when it implements correctly on three different SQL-family stacks without spec changes.

  3. 03

    Carve-outs where the world dictates.

    External protocols — Shopify HMAC, JWT claim names, callback parameter ordering — are concrete in the spec because the merchant doesn’t get to choose them. Everything else stays abstract. The discriminator is simple: who decides?

  4. 04

    Verifiable, not trustworthy.

    Every block ships an acceptance.md checklist the agent runs after implementation. Correctness comes from passing the checks, not from the spec author being clever. Trust the test, not the author.

  5. 05

    Composable through prerequisites.

    Blocks declare what they depend on, and reverse-lookup tells what depends on them. Composition becomes a graph the agent can plan around — not a configuration puzzle the developer has to solve at integration time.

  6. 06

    Living, no versioning.

    The spec is markdown, not a package. Every iteration improves it — sharper threats, better fixtures, tighter acceptance. No semver. No deprecation. The agent re-reads the latest each time it implements.

The package workaround

  • Locks framework, ORM, runtime — fits one stack
  • Customization only through config the author imagined
  • Bugs live inside the dependency — fork or wait
  • New feature = write new code from scratch
  • Ceiling = what was already written

The spec paradigm

  • Agent adapts the implementation to any SQL-family stack
  • Customization through interview — plain-language decisions
  • Bugs fixed in-place, in your codebase, by you or the agent
  • New feature = a new markdown file
  • No ceiling: better models × better specs × better tests
Why now

Agents changed the cost equation.

For forty years we shipped code packages because that’s what tooling allowed: install, configure, hope it fits. The work of translating a feature into code that fits your stack was expensive human work, so we cached it inside packages.

Claude Code reads your filesystem, runs your build, follows your conventions, iterates until tests pass. Translation is now cheap machine work. So the human job moves up the stack: write the spec, let the agent translate. The package was always a workaround for tools that couldn’t read code. We have those tools now.
