Philosophy

Don’t solve agentic problems with classical tools.

Packaging features as fixed code — whether you call them “primitives”, “components”, or “modules” — carries every classical problem forward: customization is limited to predefined config, composition breaks unpredictably, versioning multiplies. When an agent can read filesystems, run commands, and iterate, let it solve the problem directly. A blueprint is knowledge, not code.

Pre-built code old paradigm

  • Rigid — only customizable through config JSON
  • Fits one fixed runtime; merchant must match it
  • Bugs live inside the pre-built package — hard to debug
  • New feature requires writing new code from scratch
  • Surface area shrinks as integrations multiply
  • Fixed ceiling: capability equals what was written

Blueprint new paradigm

  • Flexible — the agent adapts the entire implementation
  • Fits any stack: SQL family, frameworks, runtimes
  • Bugs are fixed in-place inside the merchant’s codebase
  • New feature is a new spec — markdown, not code
  • Surface area grows with each iteration of the agent
  • No ceiling: better models × better specs × better tests

Three layers in every spec

The tension of the blueprint paradigm: too abstract and the agent has no guidance; too concrete and the spec becomes code in markdown. Every spec resolves it with three layers.

L1 — SEMANTIC

What & Why

Data model, sequence flows, state machines, business invariants, external protocol contracts, threats and mitigations.

Maximally concrete
L2 — MECHANISM

How

Framework, ORM, SQL dialect, test runner, error-handling style, file convention. The agent picks all of this.

Abstract — agent decides
L3 — ILLUSTRATIVE

Reference

Code snippets ≤30 lines, each tagged with PATTERN / PURPOSE / REFERENCE / ADAPT markers.

Concrete but marked
Workflow

How an agent consumes a block.

Claude Code — running in the merchant’s workspace with full filesystem access — treats every block as a reference document. It reads, understands the stack, adapts, and verifies.

STEP 01

Discover

Match the merchant’s request against the library by tag, category, and overview.

STEP 02

Interview

Surface configuration decisions, lock business choices, present scenarios as plain language.

STEP 03

Clone & customize

Copy the block into the project as a customized blueprint — the source library stays untouched.

STEP 04

Implement

Translate the spec into code that follows the merchant’s conventions and frameworks.

STEP 05

Verify

Run the acceptance checklist: migrations, tests, type-check, lint, security mitigations.
