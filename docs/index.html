Why a spec, not a library

Ship the spec, not the lib.

A code library locks in your framework, your ORM, your auth choice. The minute the stack diverges — fork, patch, maintain. Now agents can read your codebase, run your build, and write code that fits. So ship knowledge, not code.

Pre-built code old paradigm

  • Rigid — only customizable through config JSON
  • Fits one fixed runtime; merchant must match it
  • Bugs live inside the pre-built package — hard to debug
  • New feature requires writing new code from scratch
  • Surface area shrinks as integrations multiply
  • Fixed ceiling: capability equals what was written

Blueprint new paradigm

  • Flexible — the agent adapts the entire implementation
  • Fits any stack: SQL family, frameworks, runtimes
  • Bugs are fixed in-place inside the merchant’s codebase
  • New feature is a new spec — markdown, not code
  • Surface area grows with each iteration of the agent
  • No ceiling: better models × better specs × better tests
Workflow

How a block becomes code.

Claude Code runs in your project with full filesystem access. It treats every block as a reference — reads it, understands your stack, adapts, implements, verifies.

STEP 01

Discover

Match your request against the library by tag, category, and overview.

STEP 02

Configure

Ask the few business decisions that change behavior — currency, plan tiers, scopes. You answer in plain language.

STEP 03

Clone

Copy the block into your project as a customized blueprint — the source library stays untouched.

STEP 04

Implement

Translate the spec into code that follows your conventions and framework choices.

STEP 05

Verify

Run the acceptance checklist: migrations, tests, type-check, lint, security mitigations.

For block authors

Three layers in every spec.

Writing a new block? Every spec follows a 3-layer abstraction discipline so it stays stack-agnostic — the agent can implement it on any SQL family, any framework, any runtime, without spec changes.

L1 — SEMANTIC

What & Why

Data model, sequence flows, state machines, business invariants, external protocol contracts, threats and mitigations.

Maximally concrete
L2 — MECHANISM

How

Framework, ORM, SQL dialect, test runner, error-handling style, file convention. The agent picks all of this.

Abstract — agent decides
L3 — ILLUSTRATIVE

Reference

Code snippets ≤30 lines, each tagged with PATTERN / PURPOSE / REFERENCE / ADAPT markers.

Concrete but marked
