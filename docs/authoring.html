Page B · Spec discipline

Three layers,
one test.

A spec describes what and why, not how. The agent picks the how. A spec is at the right level of abstraction when Claude Code can implement it correctly on three different SQL-family stacks without spec changes.

Core principle
CategoryDefinitionWho decides?
WHAT Behavior, contract, invariant — what must be true regardless of stack. Domain + external party
WHY Reason the constraint exists — threat model, business rule. Spec author
HOW Framework, syntax, ORM, file structure. Merchant project (through Claude Code)

Self-test: read a paragraph of spec and ask — does this prescribe behavior, or does it prescribe how to write code? If the latter, it’s too HOW. Abstract it up.

Three layers
L1 — SEMANTIC

What & Why

Data model with logical types, sequence flows, state machines, business invariants, external protocol contracts, threats and mitigation logic, acceptance criteria.

Maximally concrete
L2 — MECHANISM

How

Framework, ORM, SQL dialect, test runner, error-handling style, file convention, naming convention at code level. The agent picks all of this from your project context.

Abstract — agent decides
L3 — ILLUSTRATIVE

Reference

Code snippets ≤30 lines, each tagged with PATTERN / PURPOSE / REFERENCE / ADAPT markers. Pattern not copy-paste.

Concrete but marked
External protocol carve-outs

Things dictated by an external party must be concrete, because the merchant doesn’t get to choose them. The discriminator: who decides?

KindExampleSpec must be
Crypto algorithm dictated by protocolHMAC-SHA256 for Shopify webhookConcrete (Shopify decides)
Crypto algorithm freely chosen“encrypt token at rest”Abstract (merchant picks AES-GCM vs ChaCha20)
JWT claim namesiss, dest, aud from App BridgeConcrete (Shopify decides)
JWT verification libraryjose vs jsonwebtokenAbstract (merchant picks)
HTTP header namesX-Shopify-Hmac-Sha256 (case-sensitive)Concrete (Shopify decides)
OAuth callback param ordering“sort params alphabetically before HMAC”Concrete (Shopify rule)
Internal job queueBullMQ vs Inngest vs PgQueueAbstract (merchant picks)

Anti-pattern: writing “verify webhook using a cryptographic hash” — too abstract; Claude Code could pick SHA-1 or MD5 and break the contract. Correct: “verify webhook using HMAC-SHA256 over the raw body, compare base64 against header X-Shopify-Hmac-Sha256 using constant-time comparison.”

The 3-stack test

One spec, three stacks, no changes.

A spec is at the right abstraction level when Claude Code implements it correctly — passes the full acceptance.md — on three different SQL-family stacks without spec changes.

StackRuntimeFrameworkDBORMTest
ANode 20ExpressPostgres 16DrizzleVitest
BBun 1.xHonoSQLiteraw SQLBun test
CDeno 2.xOakPostgres 16PrismaDeno test

NoSQL (Mongo, DynamoDB, Firestore) is explicit non-goal — documents and SQL diverge too much to derive from a single logical-types table.

Anti-patterns

× Prescribing the framework

“Add an Express middleware that…” — binds the spec to one framework.

Instead: describe the behavior. “Verify HMAC before the handler reads the body. Reject with 401 on mismatch.”

× Vague crypto

“Verify the webhook with a cryptographic hash.” — agent could pick MD5.

Instead: name the algorithm. “HMAC-SHA256 over the raw body, base64-compared against X-Shopify-Hmac-Sha256 with constant-time comparison.”

× Dialect-specific SQL

“shop_domain VARCHAR(255) UNIQUE” — MySQL-flavored, breaks on SQLite.

Instead: use logical types. “shop_domain: text UNIQUE (resolves to VARCHAR/TEXT per dialect).”

× Implementation snippets without markers

A 60-line TypeScript code block dropped in the spec with no ADAPT guidance.

Instead: ≤30 lines with all 4 markers (PATTERN / PURPOSE / REFERENCE / ADAPT) — explicit what to adapt per stack.
