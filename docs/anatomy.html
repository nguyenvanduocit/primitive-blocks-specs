Page A · Block structure

A block is
a folder.

Inside: a README defining the feature, one or more Gherkin scenarios, a security threat model, an acceptance checklist, optional fixtures. Conventions are loose; contracts are real — Claude Code reads each file for a different purpose.

Folder layout 
primitive-blocks-specs/
├── INTERFACE.md                       block interface (what counts as a block)
└── {category}/
    └── {feature-name}/                each block = one folder
        ├── README.md                  identity, overview, data model
        ├── frontend.md                optional: frontend guide
        ├── backend.md                 optional: backend guide
        ├── security.md                threat model + mitigations
        ├── *.feature                  Gherkin scenarios (BDD)
        ├── fixtures/                  test data, mock responses
        │   └── *.json
        └── acceptance.md              verification checklist
File contracts
README.md
Identity (id, name, version, category, tags), overview (problem statement, user stories), and the data model — logical tables and their semantics. The first file the agent reads.
backend.md · frontend.md
Per-side implementation guides — patterns, sequence flows, integration points. Optional but recommended for non-trivial blocks.
security.md
Threats catalogued as ### N. Name with an **Impact** severity line and mitigation logic — not code. The agent translates mitigations into your stack.
*.feature
Gherkin scenarios — happy paths, errors, edges. Each scenario is one acceptance behavior in plain English. Tags (@happy, @error, @edge) classify intent.
fixtures/
Seed data, mocked external responses, sample payloads. JSON files referenced by scenarios.
acceptance.md
The done-checklist. Sections grouped by concern (Database, API, Security, …) each with - [ ] items. The agent runs every check after implementation.
How a block becomes code

Claude Code runs in your project with full filesystem access. Five steps from spec to merged PR.

STEP 01

Discover

Match your request against the library by tag, category, and overview.

STEP 02

Configure

Ask the few business decisions that change behavior — currency, plan tiers, scopes. You answer in plain language.

STEP 03

Clone

Copy the block into your project as a customized blueprint — the source stays untouched.

STEP 04

Implement

Translate the spec into code that follows your conventions and framework choices.

STEP 05

Verify

Run the acceptance checklist: migrations, tests, type-check, lint, security mitigations.

A real block

auth.shopify-oauth, opened.

The install-handshake block every Shopify app must implement. Nine files; each does one thing.

auth/shopify-oauth/
├── README.md                       shops + oauth_nonces tables, install flow
├── backend.md                      Express/Hono/Fastify-agnostic handler patterns
├── security.md                     5 threats: CSRF, shop spoofing, code reuse, …
├── install-flow.feature            scenarios for install
├── uninstall.feature               scenarios for app/shop uninstall webhook
├── security.feature                attack scenarios with expected rejections
├── fixtures/
│   ├── oauth-callback.json
│   └── shop-records.json
└── acceptance.md                   36 checks: Database, Install, Uninstall, Security, …

Drop the block on Claude Code → it picks your SQL dialect, your framework, writes the install handler, callback, nonce store, token encryption, uninstall hook. Runs the 36 checks until green. ~45 minutes from spec to merged PR.
